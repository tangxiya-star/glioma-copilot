# 胶质母细胞瘤(GBM)共享决策证据副驾 —— 产品需求文档

## 1. 产品名称

**GBM 共享决策证据副驾(GBM Shared Decision Evidence Copilot)**

## 2. 一句话概述

一个 AI 辅助工具,帮助神经肿瘤团队快速整理经过核验的临床试验证据、用通俗语言向患者解释治疗选项,并结构化地记录患者意愿,以支持复杂胶质母细胞瘤诊疗中的医患共享决策。

---

## 2.5 概要:解决什么问题、用什么方法、数据从哪来

> 本节是全文速览。若时间有限,读这一节即可把握全貌。

### ① 在解决什么问题(Problem Statement)

复发性胶质母细胞瘤(GBM)患者的常规治疗选择极为有限,临床试验因此成为关键出路。但医生要为某位患者找到合适的试验,必须手工把散落在多处的信息拼起来:试验登记库(ClinicalTrials.gov)、患者的分子标志物(IDH、MGMT、EGFR 等)、药物的监管审批信息、以及大量文献。更难的是,试验的**入排标准**用冗长复杂的语言书写,而**患者意愿**(能否长途就医、看重生活质量还是积极治疗等)通常只停留在口头,没有被结构化记录。

**结果**:证据审阅、患者沟通、决策留档三者之间存在断层——过程耗时、难核验、且难以在诊疗当下保持实时。

### ② 用什么方法解决(Approach)

用一个以大语言模型(Claude)为核心、但**以权威数据源为事实依据**的工作流,把上述断层缝合起来:

1. **检索**:根据患者信息,从 ClinicalTrials.gov 实时检索相关试验。
2. **匹配 + 逐条判定**:让模型阅读每项试验的入排标准原文,对照患者情况逐条判断"符合 / 不符合 / 信息缺失",而非简单关键词匹配。
3. **核验(核心创新)**:采用多 Agent 相互校验的闭环——一个 Agent 起草证据结论,另一个 Agent 拿原始记录逐条核对、抓出并改写"过度声明"(例如把"该患者符合试验 B"改写为"可能相关,但需先补测 EGFR")。
4. **通俗化 + 意愿采集**:把医学术语翻译成患者能懂的解释,并结构化记录患者意愿。
5. **可溯源输出**:生成一份共享决策简报,其中**每一条临床声明都链接回其来源**(试验记录、病理报告等)。

**关键原则**:模型不是"真相来源",它只在检索到的权威记录之上做推理和表达;所有事实都可追溯、可核验。

### ③ 数据从哪来(Data Sources)

| 数据 | 来源 | 用途 |
|------|------|------|
| 临床试验(标题、分期、状态、入排标准、地点等) | **ClinicalTrials.gov / AACT** | 试验检索与匹配的核心数据 |
| 药物名称标准化 | **RxNorm / ChEMBL** | 把杂乱药名(商品名/缩写/剂型)统一到标准药物与 ID |
| 监管信息(审批状态、适应症、说明书) | **FDA(openFDA)** | 为药物证据提供监管背书 |
| 分子标志物(IDH/MGMT/EGFR 等) | 从试验入排标准 + 病理报告中提取 | 把试验要求与患者分子谱匹配 |
| 患者病例 | **合成模拟患者**(隐私合规) | 演示用,避免真实患者隐私问题 |

## 3. 背景

复发性胶质母细胞瘤(GBM)患者的标准治疗选择通常非常有限,因此**临床试验**成为其诊疗中的关键一环。然而,为患者筛选合适的试验,需要临床医生手工检索并整合分散在多处的证据:

- **ClinicalTrials.gov**(全球临床试验登记库)上的试验信息;
- **分子标志物**(如 IDH、MGMT 甲基化状态、EGFR 扩增、TERT 突变等);
- **FDA 等监管文件**中的药物审批与适应症信息;
- 快速更新的**文献**。

试验筛选并不只是医学层面的入排标准判断。临床医生还需综合考虑患者年龄、体能状态(如 ECOG/KPS)、既往治疗、地理位置、照护者支持、随访频率、出行意愿以及患者的治疗目标。

这一过程耗时、难以核验,且在临床决策的当下很难保持信息实时更新。

---

## 4. 待解决的问题

当前复杂 GBM 诊疗中,临床试验评估的工作流是碎片化的:

- 试验数据在 ClinicalTrials.gov;
- 药物与分子标志物证据散落在文献、FDA 文件和各机构的内部知识里;
- 试验入排标准以冗长、复杂的语言书写;
- 患者意愿往往只在口头讨论中提及,没有被结构化记录;
- 医生需要在情绪沉重的医患沟通中,把复杂的试验术语翻译成患者能理解的解释。

这在以下三者之间造成了断层:

1. 证据审阅
2. 患者沟通
3. 共享决策的记录留档

---

## 5. 目标用户

### 主要用户:神经肿瘤科医生 / 肿瘤临床医生
需要在与患者讨论治疗方案前,快速审阅相关试验与证据。

### 次要用户:患者及照护者
需要用通俗语言理解试验选项,并就出行、风险承受度、生活质量、照护者支持、后勤安排等表达意愿。

### 支持用户:临床试验协调员(Clinical Trial Coordinator)
协助筛查入排标准、确认试验可及性,并与试验中心跟进对接。

---

## 6. 核心使用场景

一位复发性 GBM 患者正在神经肿瘤团队处就诊。临床医生希望:识别出相关的临床试验、理解每项试验为何可能适合或不适合、向患者清晰解释这些选项,并记录患者的意愿与后续步骤。

---

## 7. 产品目标

1. 帮助临床医生更快地整理试验证据。
2. 让患者能够理解试验选项。
3. 结构化地捕捉患者意愿。
4. 支持共享决策,但**不取代**临床判断。
5. 确保每一条声明都有来源支撑、可核验。

---

## 8. 非目标(明确不做的事)

本产品**不会**:

- 独立推荐某项治疗方案;
- 取代临床医生的判断;
- 预测生存期;
- 向患者提供直接的医疗建议;
- 自动为患者报名参加试验;
- 把自由文本的临床理由归类进缺乏依据的评分系统。

---

## 9. 产品定位

这不是一个治疗推荐工具。

它是一个面向复杂神经肿瘤诊疗的**「经核验的证据 + 共享决策支持层」**。

目标是帮助医生与患者共同理解证据、揭示现实层面的约束条件、并记录试验讨论背后的推理过程。

> **定位陈述**:我们不帮助 AI 去"选择"临床试验。我们帮助医生和患者共同理解证据、呈现患者意愿,并记录共享决策的过程。

---

## 10. 用户工作流

### 第一步:临床医生工作台

医生录入或导入患者信息:

- 诊断:胶质母细胞瘤
- 疾病状态:初诊 / 复发
- 年龄、地理位置、既往治疗
- 分子标志物:IDH、MGMT、EGFR、TERT
- 体能状态(若有)
- 出行约束(若已知)

系统检索并输出:

- 相关临床试验及其状态、入排标准、相关药物、标志物要求、监管历史(若有)
- 证据冲突或不确定之处
- **候选试验清单 + 入排理由 + 证据简报 + 来源引用 + 核验警示**

### 第二步:共享决策工作台(医患共同使用)

对每项试验,系统分别呈现:

**医生视图**:试验名称、分期、地点、入排理由、所需标志物匹配、既往治疗要求、随访频率、证据摘要、风险/不确定性、来源链接。

**患者友好视图**(通俗语言解释):这项试验在测试什么、患者为何可能符合、参与可能涉及什么、出行/就诊负担、已知与未知、可向诊疗团队提出的问题。

> 示例:"这项研究正在测试一种针对特定分子标志物肿瘤患者的实验性疗法。您的病理报告提示您可能匹配该标志物。该试验可能需要定期就诊和额外的影像检查。它不保证一定有效,但可能提供一种其他途径无法获得的治疗机会。"

### 第三步:患者意愿采集

患者或照护者回答结构化问题:是否愿意跨州/跨地区就医、能多频繁到院、当前更看重生活质量还是积极治疗、是否接受早期(如 I 期)实验性试验、是否有照护者支持、费用/住宿/医保是否是重要顾虑、是否愿意在需要时接受再次活检等。

这些意愿不用于做出最终决定,而是帮助医生理解现实层面的试验适配度。

### 第四步:决策讨论摘要

系统生成一份共享决策摘要,包含:讨论过的试验、每项为何可能适合/不适合、患者表达的意愿、待解决的问题、后续步骤。

---

## 11. 核心功能

- **11.1 患者病例录入**:诊断、标志物、治疗、地点、意愿的结构化表单;支持导入模拟病例用于演示。
- **11.2 证据图谱**:存储试验、药物、标志物、申办方、地点、监管记录、试验状态之间的关联数据。
- **11.3 试验匹配**:基于疾病、复发状态、标志物、既往治疗、年龄、地点、试验状态、出行意愿进行匹配。
- **11.4 证据核验(三-Agent 工作流)**:
  - *起草 Agent* —— 生成有来源支撑的证据简报;
  - *核验 Agent* —— 逐条核对声明,标记缺乏依据或含糊的声明;
  - *调查 Agent* —— 检索更多记录或收窄无依据的声明。
- **11.5 通俗语言解释**:把面向医生的试验术语翻译成患者能懂的解释。
- **11.6 意愿感知的试验审阅**:根据患者意愿(地点、出行、随访频率、风险承受度)对试验重新排序或标注。
- **11.7 共享决策记录**:生成经医生审阅的摘要,适合归档留档。

---

## 12. 数据来源

- **12.1 临床试验数据** —— 来源:ClinicalTrials.gov API / AACT。用于:试验标题、状态、分期、地点、申办方、入排标准、干预措施、适应症。
- **12.2 药物名称归一化** —— 来源:RxNorm + ChEMBL。用于:把杂乱的药名(商品名、缩写、含剂型的完整名)映射到标准药名及标准 ID。例:Temodar / TMZ / "替莫唑胺 150mg 胶囊" → 标准药物:替莫唑胺(Temozolomide)。
- **12.3 监管证据** —— 来源:FDA(Drugs@FDA / openFDA)。用于:FDA 标签、适应症、审批状态、监管历史。
- **12.4 分子/标志物数据** —— 来源:从试验入排标准 + 模拟患者病理报告中提取。用于:把试验标准与 GBM 分子谱匹配(IDH 野生型/突变、MGMT 甲基化/未甲基化、EGFR 扩增、TERT 突变、1p/19q 共缺失等)。
- **12.5 患者病例数据** —— 出于隐私合规,演示阶段使用**合成模拟患者**(年龄、诊断、疾病状态、地点、既往治疗、分子标志物、体能状态、出行意愿、照护者支持、治疗目标等字段)。

## 13. 技术架构

- **前端**:Next.js / React、Tailwind CSS、shadcn/ui(临床工作台、共享决策工作台、意愿表单、试验卡片、证据引用)。
- **后端**:FastAPI(Python)、Pydantic(API 接口、试验检索、证据组装、Agent 编排)。
- **数据/图**:Neo4j(Cypher)存储药物-试验-标志物-申办方-地点-监管记录之间的关系;对于轻量实现也可用 PostgreSQL 替代。
- **大语言模型 / Agents**:Claude API,承担起草、核验、调查、通俗化解释、共享决策摘要等 Agent。
  > **重要原则**:Claude 不是真相来源。Claude 只在检索到的来源记录之上做推理。
- **数据处理**:Python、pandas、requests(ClinicalTrials.gov 摄取、药名归一化、监管数据解析、入排标准文本抽取)。

### 高层架构

```
ClinicalTrials.gov / AACT
        │
   试验摄取 + 入排标准解析
        │
RxNorm / ChEMBL ──► 标准药物解析层 ◄── ChEMBL 药物数据
        │
     关系存储(Neo4j 或 PostgreSQL)
     药物-试验-标志物 / FDA-地点-申办方
        │
     FastAPI 后端
        │
  ┌─────┼─────────────┐
  ▼     ▼             ▼
起草Agent  核验Agent   解释Agent
  └─────┬──┴────┬──────┘
        ▼       ▼
   临床工作台  共享决策工作台
```

## 14. 风险与缓解

- **临床风险**:产品可能被误解为在做医疗推荐。→ 缓解:统一使用"供临床医生审阅的证据""讨论支持""可能相关的试验"等措辞,避免"推荐的治疗"。
- **证据风险**:试验数据可能陈旧或不完整。→ 缓解:展示来源、最后更新日期和不确定状态。
- **患者沟通风险**:通俗化解释可能过度简化不确定性。→ 缓解:始终包含"已知什么、未知什么、可向医生提出哪些问题"。

## 15. MVP 范围

- 单一病种:复发性 GBM;
- 一到两个模拟患者病例;
- ClinicalTrials.gov 试验摄取;
- 基础的药物/标志物归一化;
- 轻量图结构(Neo4j 或 PostgreSQL);
- 三-Agent 核验闭环;
- 两个界面:临床医生证据工作台 + 共享决策工作台。

---

## 16. 产品愿景陈述

**GBM 共享决策证据副驾**帮助神经肿瘤团队,把碎片化的试验数据、分子证据和患者意愿,转化为一份经核验、有来源支撑的共享决策简报。

它不取代临床医生的判断。它帮助医生与患者共同理解复杂的试验选项。
